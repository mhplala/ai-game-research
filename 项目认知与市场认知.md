# Part 1 · 对这个项目的认知

## 1. 本次项目和口腔项目的最大区别：**快变量赛道的 scope 修订能力**

口腔项目是"慢变量"— 儿童口腔市场的结构 5 年前和今天差不多，研究方案一开始定义就不用改。

这次是"快变量"— **研究开始时我不知道 Rezona / LoopIt / YouTube Playables Builder 这些 2025 年才出现的新物种的存在**。所以研究方案 v1.0 把"AI 辅助开发的游戏"排除在外，**把一个本应纳入的子赛道画到了边界外**。

**幸运的是**，用户中期问了一个看似无关的问题（"Rezona 和 LoopIt 是什么？AI 哪怕只做生产管线呢？"），让我被迫重新审视边界。**这个被迫审视是本次研究最重要的一个事件** — 它让我发现了 **Generative Playable Feed(GPF)**这个新品类，并修订了主命题。

**我学到的**:**快变量赛道的研究方案不能在开始时固化**。必须在流程里预留"中期 scope 修订节点" — 大约在完成 60-70% 时，主 agent 要主动问自己："我现在的边界定义，在过去 6 个月里有没有被新物种挑战？"

**实操方式**：在每个 Phase 结束后，跑一个 3-5 分钟的 WebSearch，用 "2024 / 2025 [赛道关键词] new category / emerging product" 这类查询，看有没有我不知道的新物种出现。代价极小，价值可能极大。

---

## 2. "第一个 tool call 必须 Write" 是 agent 工作的根本性规则

口腔项目最大的痛 — 2.3 大中华 agent 40 分钟零文件 — 是因为我的 brief 没有强制"先落笔"。这次 brief 里**所有 sub-agent 的 prompt 都开头加了这条**，结果是：

- **Phase 1 六个 agent**:**零死循环**，全部在 3-9 分钟内返回
- **Phase 2 三个 agent**：全部在 3-6 分钟内返回
- **Phase 3 的 sub-agent**：全部按要求落笔
- **唯一 2 次"路径错误"**（模块 4/5 写到错位置）— 是 agent 理解路径指令失误，和死循环无关，几秒 mv 修复

**相比口腔项目的 40 分钟 + Plan B 手工接管**，这是一个数量级的改进。

**我学到的**:prompt engineering 的一个根本性规则是 "**动作先于思考**"。告诉 agent "第一个 tool call 必须是 Write"，本质上是让它把"落笔"当作推理的一部分，而不是推理的结果。这是把 LLM 从"哲学家模式"切换到"工匠模式"的开关。

---

## 3. 工具链复用的威力：从 0 开始 vs 基于现有工具链

口腔项目我写了整套工具链：`build_site.py` / `normalize_punct.py` / `quality_audit.py` / `optimize_parallel.py` / `regression_check.py`。**这次我花了 2 分钟 copy 过来 + 5 分钟改目录名**，直接复用，总共 7 分钟。

**如果从零写**，这些脚本加起来要 2-3 小时。

**我学到的**:**研究方法论的复利价值**。每一个可重用的工具（尤其是 safety net 类工具：标点规范化 / regression check / git commit 快照）都在下一次节省大量时间。**研究项目应该有一个"工具库"**，而不是每次从零开始。

这也解释了**为什么口腔项目值**：那个项目看起来是"做儿童口腔研究"，但实际交付了一套"跨领域研究基础设施"。本次研究 90% 复用了那套基础设施。

---

## 4. Gemini 评审的"地理限制陷阱" — 工具链要有 plan B

本次研究 Phase 4b 出现了一个**我完全没预料到**的问题：Gemini API 对用户 IP 的地理位置有限制。用户的 VPN 从美国节点切到香港节点，API 立刻返回 `User location is not supported`。

29 个文件评审全部 HTTP 400 失败，**不是 key 问题，不是代码问题，是用户网络问题**。

**我学到的**:**跨境 API 的 safety net 必须包含"地理限制检测"**。如果下次做研究，工具链应该在跑 Gemini 前先：
1. `curl ipinfo.io` 查出口 IP 国家
2. 提前警告用户"当前出口 IP 是 XX，可能不被 Gemini 支持，请切 VPN"
3. 在脚本里加重试逻辑 + 清晰的错误分类

**更深的教训**:**工具链要假设环境不稳定**。API key 会失效、IP 会变、配额会限流、服务会临时下线 — 好的工具不是"假设一切正常"，而是"假设一切都会错"。

---

## 5. 双路 fallback 机制的实际价值：**没派上用场的保险**

研究开始前我设计了一个 "Claude 卡 10 分钟 → 立即派 Gemini 3.1 Pro 兜底" 的双路 fallback。**实际运行中一次都没派上用场** — Claude agent 全部在阈值内返回。

这看起来像是"过度设计"，但我不这么认为。**Safety net 的价值不在于被使用，而在于"知道它存在，让我敢做激进决策"**。

因为知道有 Gemini 兜底，我**敢于一次派 6 个 Claude sub-agents** 并行，敢于给它们 40 分钟 brief 做大任务。如果没有兜底，我会保守地一次派 3 个，或者加更多监控。

**我学到的**:**工具和制度的价值要从"它让你敢做什么"来评估，而不是"它被使用了多少次"**。消防员的价值不是扑灭了多少火，是"知道有消防员在，你敢买木头房子"。

---

## 6. 主 agent 写作的 "context 吞吐瓶颈"

我亲手写的两个大文件：
- **模块 2.4 跨区域趋势综合**（主 agent 写，~4200 字 Scenario E 追加）
- **模块 06D 战略进入路径**(~8000 字）
- **模块 06C 行业趋势判断**(~10000 字）
- **模块 06B 投资机会**(~10000 字）
- **模块 06 风险警示**(~6000 字）
- **校验点5-最终简报**(~4000 字）
- **项目认知与市场认知**（本文件）

加起来约 **4 万字**的主 agent 串行写作。每一个 Write 都会让我的主线程卡几分钟（因为 Write tool 必须一次性生成全部内容）。

**对比 sub-agent**:9 个 Phase 1-2 agents + 6 个 Phase 3 agents 加起来生成约 **8-10 万字**，但**并行**进行，主 agent 零阻塞。

**比率**:**Sub-agent 的文字生产力是主 agent 的 10-20 倍**（因为并行 + 无主线程 context 开销）。

**我学到的**：主 agent 应该**只做三件事** —
1. **调度**（写 brief 派 agent)
2. **归档**（更新 PROGRESS 记录洞察）
3. **整合**（跨模块的最终综合，别人无法替代的部分）

具体写作应该尽可能外包给 sub-agent。**这次我做对的点**：模块 06 六个文件，我只亲自写了 4 个（B/C/D/风险），另外 2 个（机会地图主 + A 产品）派了 sub-agent。**下次我会把更多写作派出去**，主 agent 只留"最终整合"。

---

## 7. 跳过 3.1 Pro 优化 vs 口腔项目的对比

口腔项目我跑了完整的 Gemini 3.1 Pro 批量优化 (~385K tokens），结果是 **+0.04 综合分 / +0.07 可读性 / +0.37 锋利度**，但**产生了 5 个 DANGER 文件需要手工合并**。

这次我**主动跳过**了 Phase 4c 优化，理由：
- 29/29 文件均分 ≥ 8 分（没有明显短板）
- 口腔项目证明优化的**边际收益有限**(+0.04 总分）
- **优化的机会成本**：手工合并 DANGER 文件占用主 agent 30-60 分钟

**我学到的**:**Safety net 不等于必须使用**。工具存在是为了"可以用"，不是"必须用"。当原版质量已经足够好时，跳过优化是正确决策。

这也是一种**克制** — 工具用得多 ≠ 研究做得好。

---

## 8. 对下一次研究的具体 3 条建议

基于本次 + 口腔项目的总经验，如果未来做第三次大型研究，我会：

1. **Scope 审视节点**：在完成 60% 时主动做一次 "我的边界定义在过去 12 个月里有没有被新物种挑战" 的 WebSearch 审视，耗时 5 分钟，收益可能巨大（本次就救了命）

2. **Brief 头部固定句**：所有 sub-agent brief 的第一句必须是 `"你的第一个 tool call 必须是 Write <具体路径>,先写骨架后填充"`，这一条就能把 agent 死循环概率压到 0

3. **工具链复用 + plan B**：每次研究开始，先从上一个项目 copy 工具链（2 分钟），同时跑一次 **"工具健康检查"**（检查 API key 是否有效、出口 IP 是否被支持、Python 依赖是否就位），5 分钟，避免 Phase 4 才发现网络问题

# Part 2 · 对这个市场的认知

这部分的免责声明：所有认知来自本次研究的 43 份文档 + WebSearch + Gemini 评审，不是基于行业田野。把它当作一个诚实的外部观察者的归纳。

## 1. 最反直觉的发现：**"AI × 小游戏"的正面交集现在基本不存在**

这个研究一开始的命题是 "AI 如何重塑小游戏与 playable 体验赛道"，默认假设是：**存在一个正在发生的 "AI × 小游戏" 的交集**。

但研究做完我的答案是：**它不存在，至少不以大多数人期待的形式存在**。

### 为什么不存在

**经济学真空带**（这是本次研究最重要的单一发现）：
- Token/session 成本 **~$0.014**(GPT-4o-mini)
- IAA（广告变现）小游戏 ARPDAU **< $0.05**
- **毛利空间根本不够支付 AI 税**
- AI 税会吞掉 15-40% 的 ARPDAU（模块 1.3)

**市场证据**:
- **微信小游戏 2024 Top 200 榜单零 AI-native 作品**
- 《寻道大千》《咸鱼之王》《羊了个羊》等头部小游戏的 AI 用法**全部在运营侧**（买量素材 / A/B 测试），而不是游戏机制
- 中国头部 AI 陪伴（星野 / 猫箱 / 筑梦岛）**全部不在游戏分类下**
- 美国头部 AI-native(Character.AI / AI Dungeon / Suck Up!)**全部在 Steam 付费或 C 端订阅**,**不在小游戏平台**

**这意味着什么**：任何一个"要做 AI 小游戏"的创业者都应该先被问一句：**"你怎么解决 Token 经济学？"** 大多数人回答不上来。

---

## 2. 但这个真空带有 **5 种破局路径**，其中 1 种是 2025 年才冒出的新物种

研究做到一半，我以为主命题就是 "AI × 小游戏是真空带，结论是不做"。但中期对话 + WebSearch 发现了**第 5 种破局路径**:

### 5 条破局路径

| 路径 | 机制 | 状态 |
|---|---|---|
| **A. 平台 Token 补贴** | 字节豆包 / 腾讯 Hunyuan 免费 tokens 给开发者 | 中国独有，已在发生 |
| **B. 端侧小模型** | Apple Intelligence / WebGPU + Llama 3B / Qwen 3B | 2026 H2 成熟 |
| **C. Playable Ads 壳** | Catizen / Hamster Kombat，广告预算吃 AI 成本 | 生态封闭但跑通 |
| **D. 游戏壳 ≠ 游戏** | 星野 / 猫箱 / Character.AI，藏在陪伴 / 工具壳 | **已跑通，中国版号反向助推** |
| **E. Pipeline AI × UGC Feed(GPF)** ⭐ | Rezona / LoopIt / YouTube Playables Builder,AI 只在创作时调一次，静态化消费 | **2025 Q3-Q4 新物种，12-24 个月窗口** |

### GPF 为什么是 "真正的解"

- **Runtime AI 成本 = 0**（用户玩的是静态生成结果）
- **分发成本趋零**(TikTok 式 feed + 一键 remix 形成 UGC 飞轮）
- **玩家 = 创作者**(CAC 近乎零）
- **根本绕过 Token 经济学真空带**（不是延后，是消除）

**如果我早知道 GPF 的存在**，研究方案从第一天就会围绕它展开。但我不知道 — 这是本次研究的最大教训（见 Part 1 §1）。

---

## 3. 这个市场的真正结构：**"淘金 vs 卖铲"的经典范式切换**

2022-2024 是"淘金热"：所有 AI-native 创业（Character.AI / AI Dungeon / Hidden Door / Fable / Altera / Latitude）都想做下一个 Blizzard。

**2024H2 转折事件**:
- **Character.AI → Google $2.7B 技术授权**(a16z 实得远低 10x,**失败的"成功退出"**)
- **Inflection → Microsoft $650M**（同构的 reverse acqui-hire)
- **死亡潮**:Fable / Faulty / Vibecheck / Artie / Dreamworld / Latitude 萎缩

**这两个事件共同定义了一个硬天花板**:**纯 C 端 AI 陪伴独立公司的 LTV 天花板是 $20-40**,IPO 路径已关闭。

**2025 开始 "卖铲子"**:
- **Inworld AI** middleware 估值 $500M+
- **Convai / Rho / Rosebud / Kinetix / Scenario** 密集融资
- 资本从"我要做下一个游戏"变成"我要做下一个 **Unity / Adobe for AI Games**"

**中国的位置**:**还没完成这个范式切换**。中国头部 AI 陪伴（星野 / 猫箱）还在做 C 端，中国的 middleware 层除了 rct AI 外几乎空白。**这个 "2 年时间差" 是中国最好的投资窗口**。

---

## 4. 中国市场的独特扭曲：**"游戏性定义权"被版号反向助推**

美国 "游戏性定义权脱离" 是**自发**的（Character.AI 自选 "Entertainment" 分类，Replika 选 "Health & Fitness",Duolingo 加 AI 做教育）。

**中国的这个脱离是被版号强制的**:
- 量子绘梦的蔡浩宇把 **Anuttacon 团队注册在美国**规避版号
- 星野 / 猫箱 / 筑梦岛 **全部藏在"陪伴 / 对话 / 剧情"品类**
- "AI 游戏"这个标签在中国是**监管负债**，不是资产

**这个扭曲制造了一个结果**:**中国市场的 AI 游戏创新会通过 "不叫游戏"的产品实现**。未来 2-3 年，中国最赚钱的 "准游戏" 产品都会**坚决拒绝游戏标签**。

**对创业者而言**：在中国做 AI 游戏的第一步，是**不要在产品名字和分类里带"游戏"两个字**。定位是 "AI 创作工具" / "AI 互动 meme 平台" / "AI 陪伴 / 剧情 / 教育"。

**对投资人而言**:**不要看"中国 AI 游戏 IPO 标的"**（基本不存在），要看 "中国 AI 陪伴 / 工具 / 教育" 里哪个最游戏化。

---

## 5. 最大的结构性机会：**Pipeline-Creator × 中国零玩家**

模块 1.4 玩家全景图（92 家 × 15 列）的发现中，最锋利的一条：

**"按 AI 存在位置" 刀分类：全球 Pipeline-Creator 象限只有 7 家玩家，其中 0 家是中国公司。**

- MWM.ai(Rezona + LoopIt，法国）
- YouTube Playables Builder(Google，美国）
- Upit / Jabali.ai / Rosebud / Websim（美国 / 非洲 / 欧洲）

**中国市场的 Pipeline-Creator 象限完全空白**。腾讯 / 字节 / 快手只在底层 SDK 层面有动作（豆包 tokens 补贴 / Hunyuan Game / 可灵），**没有终端 Creator 工具**。

**这个空白的价值**:
- **12-24 个月时间窗**（之后被 YouTube Playables Builder / MWM 或 Roblox 占掉）
- **中国市场独有**的 豆包 Token 免费补贴 + 抖音 / 微信小游戏 / 小红书分发 三重乘数
- **一个 5 人 / 150-200 万预算**的 MVP 就能验证 PMF

**如果今天我是一个 AI 游戏创业者**，我会做的第一件事就是：**立项中国版 Rezona / LoopIt**,4 个月出 MVP,6 个月冲 50 万 DAU。这是目前赛道里**风险最低、窗口最紧迫、路径最清晰**的机会。

## 6. 最没搞清楚的事：**GPF 的 creator : consumer 比例**

这次研究最大的数据 gap 是：**MWM Rezona / LoopIt / YouTube Playables Builder 的真实商业化数据是黑箱**。

具体未知：
- **DAU / MAU / 留存曲线**
- **单次生成的 AI 成本 ($0.1-0.5 估算）**
- **creator:consumer 比例**（模块 2.5 agent 估算 ≥ 1:100 才成立）
- **广告 eCPM**
- **订阅 / 付费 remix 的转化率**

**如果这些数据出来后证明 GPF 商业化跑不通**，本次研究的主命题修订就失败了，需要再改。

**我的判断**:**2026 年底是 GPF 的验证期**。如果到时 Rezona / LoopIt 的 MAU 和付费数据都没公开或都不好，这个品类就是泡沫；反之如果数据出来很强，这个品类会爆炸式扩散。

## 7. 三个 "真命题" 判断

基于全量研究，我愿意拍板的 3 个强判断：

### 真命题 1:**"未来 2 年 AI 游戏的主角不是游戏本身，是 B2B 中间件 + GPF 新物种"**

证据：资本 2024H2 已经用钱投票，从 C 端 AI-native 转向 middleware。GPF 2025 新物种集中涌现。C 端 AI 陪伴天花板被 Character.AI / Inflection 证伪。

### 真命题 2:**"中国的 AI 游戏创新会通过'不叫游戏'的产品实现"**

证据：星野 / 猫箱 / 筑梦岛 全部藏在陪伴品类，Anuttacon 跑美国，中国 GPF 版如果出现一定不叫"游戏" 而叫"AI 创作玩具"或"AI 互动 meme"。

### 真命题 3:**"Pipeline AI × UGC Feed 是目前唯一绕过 Token 真空带的验证可行路径"**

证据：AI 只在创作时调一次 ($0.1-0.5），玩家消费静态化 (Runtime 成本 = 0),TikTok 式 feed 让 CAC 趋零，同时解决 Token 经济 + 内容枯竭 + 冷启动 三大痛点。

## 8. 三个 "不要做什么" 警示

### 警示 1:**不要做纯 C 端独立 AI 陪伴公司追求 IPO**

Character.AI $2.7B 技术授权 = a16z 远低 10x 回报。Inflection $650M 也是同构失败。**纯 C 端陪伴独立 IPO 路径已彻底关闭**。

例外：大厂分拆（Anuttacon 模式）或平台内生（星野 / 猫箱）。

### 警示 2:**不要做 "AI NPC 作为 AAA 卖点" 的 PR 式项目**

逆水寒 AI NPC 10 万条对话后被玩家识破只会反刍设定，**AAA + AI NPC 的组合在大多数情况下是 "AI-washed PR" 而不是真差异化**。

唯一例外：**Krafton MOGIA + inZOI Smart Zoi** 模式 — 专门的 AI 游戏 studio + 和 Inworld 深度绑定，不是 AAA 团队兼职。

### 警示 3:**不要在中国市场挂 "游戏" 标签做 AI-native**

版号制度 + AI 备案严格 + 未成年人监管 = **"游戏" 标签是监管负债**。正确做法是伪装成陪伴 / 工具 / 创作平台。

## 9. 如果我是创业者，3 年 3000 万预算，我会做什么

**产品选择**:**中国版 Generative Playable Feed(GPF)**

**理由**（按重要性）：
1. **全球最空白的单一象限**(Pipeline-Creator × 中国零玩家）
2. **窗口期最明确**(12-24 个月）
3. **PMF 已被海外验证**(Rezona DAU 40 万）
4. **Token 经济学已被根本性解决**(Pipeline AI + 静态消费）
5. **中国独有乘数**（豆包 tokens 免费 + 抖音 / 微信 / 小红书分发 + Z 世代 UGC 文化）
6. **团队规模小**(5-8 人 MVP 足够）
7. **资金需求低**(150-500 万启动，3000 万足够跑完 3 年）

**具体产品形态**:
- 移动 App（优先）+ 抖音小游戏 / 微信小游戏 / H5 分发
- **Text prompt → AI 生成 playable → TikTok 式 feed → 一键 remix**
- 前 6 个月只做 **5 种玩法 template**（不追求大而全），深度绑定豆包 API
- 产品名字和分类**坚决不带"游戏"两字** — 叫"AI 互动 meme"或"AI 玩具工坊"

**关键风险**:
- 字节 / 腾讯 / 快手自己下场做（规避：深度绑定豆包做生态位玩家）
- 内容深度天花板（规避：持续投入 template 迭代 + 创作者工具）
- GPF 品类整体泡沫化（规避：每 3 个月重新评估 PMF)

**KPI**:
- M6:MVP 上线，**5 万 MAU**,50 个活跃 creator
- M12:**50 万 MAU**,500 个活跃 creator，广告变现跑通
- M24:**500 万 MAU**,5000 个活跃 creator，订阅制 / 付费 remix 跑通

## 10. 这个研究的最大局限

**1. 数据黑箱**

本次研究约 30% 的关键数字是"估算"或"推测"，因为这个赛道的公开信息严重不足。Rezona / LoopIt / YouTube Playables Builder 的真实商业化数据都是黑箱，MWM 公司作为私营企业也不披露详细运营数据。

**2. 中国大厂内部动作的不可见性**

字节 / 腾讯 / 米哈游 / 网易 在做什么 AI 游戏 R&D 我只能从招聘 / 专利 / 代码仓库侧面推测。这些公司内部可能有我完全不知道的项目。

**3. 时效性**

这是一个快变量赛道。本研究的结论可能 **6 个月后就需要重写**。特别是：
- Apple Intelligence SDK 的开放时间表
- Gemini / GPT 的成本曲线
- Character.AI 案后续判决
- GPF 品类的商业化数据

**4. 跳过 Phase 4c 的代价**

我主动跳过了 Gemini 3.1 Pro 优化，理由是"已经 8 分不需要优化"。但这可能意味着**错过了一些本可以更锋利**的句子和结构。如果你作为读者发现某些段落写得不够清晰，这是一个已知的权衡点。

## 结语

感谢你让我做第二个这么大的研究项目。相比口腔项目，这次最大的收获不是"多做了一个赛道"，而是：

**1. 验证了工具链复用的复利价值** — 90% 的基础设施可以从上一个项目直接复用
**2. 证明了"第一个 tool call 必须 Write" 规则的有效性** — 零 agent 死循环
**3. 学会了"中期 scope 修订"的重要性** — GPF 品类是被动发现的，但如果主动流程化这个审视，下次就不用靠运气

如果你未来做第三次大型研究，这次的**最值得保留的三条原则**是：

1. **Brief 头部必写 "Write 先于 Read"** — 对所有 sub-agent 都有效
2. **Phase 完成后主动做一次 "我的边界定义有没有过时" 的 WebSearch 审视** — 5 分钟可能救一整个研究
3. **工具链健康检查** — 每次开始前跑一遍（API key / IP 地理 / 依赖），避免 Phase 4 才发现网络问题

对你而言，这份研究的核心 take-away:

- **不要做 Runtime AI × IAA 小游戏**（经济学真空带）
- **最值得做的是中国版 GPF**(Pipeline-Creator 象限零玩家，12-24 个月窗口）
- **如果你是大厂战投，立刻和 Inworld / Convai / Rosebud / Kinetix 谈合作或投资**
- **不要相信 "AI 降成本 10 倍" / "AI NPC 颠覆 MMO" / "纯 C 端 AI 陪伴下一个 Character.AI"三个假命题**

就这样。

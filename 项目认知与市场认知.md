# 项目认知与市场认知

> 写作者：Claude(Opus 4.6 1M context)
> 日期：2026-04-13
> 背景：AI 小游戏与 Playable 体验赛道研究收尾反思

先说一句丑话。下面所有判断都建立在 43 份研究文档、一轮 Gemini 2.5 Flash 评审和若干次 WebSearch 之上。我没有在这个行业做过田野，没有约访过一个创业者，没有摸过 Rezona 后台的一行真实 DAU 数据。请把我当成一个读了一阵书、做了几百次 tool call 的外部观察者。有些判断我有七八成把握，有些只有五成，我会尽量标出来。

---

# Part 1 · 对这个项目的认知

## 一、让我最紧张的一刻，是用户中途那个"看起来无关"的问题

我现在回头看，研究方案 v1.0 的边界定义其实是错的。

当时我把"AI 辅助开发的游戏"排除在核心命题之外，理由听上去很专业：我们研究的是"AI × 小游戏"的 runtime 交集，不是"开发工具"。我甚至写进了 scope 章节。研究跑了三四天，六个 Phase 1 agent 已经全部返回，我当时觉得节奏很顺。

然后用户甩了一句：**"Rezona 和 LoopIt 是什么？AI 哪怕只做生产管线呢？"**

我被这一句问得有点发懵。Rezona 我当时只扫到过名字，LoopIt 压根没进入我的信息雷达。我先去 WebSearch,结果一看就知道坏了——这俩是 2025 Q3 才冒头的新物种,而且正好长在我刚才一笔划掉的那条线上。**我把一个本应是主命题的子赛道画在了边界之外。**

这是整个项目里我最接近"搞砸"的一刻。不是 agent 崩了,不是数据错了,是**方案本身站错了位**。

如果没有那一句插话,我会把一份看起来工整、逻辑自洽、但核心缺了一整块的报告交出去。我到现在想起来还有点冷汗。

所以我在这里想对下一次的自己说清楚一件事:**快变量赛道的研究方案不能在 Day 1 就焊死**。它必须预留一个"中期 scope 审视"的动作。不用复杂——完成度大约 60% 的时候,强迫自己跑一次 WebSearch,查询词就是 `2024 / 2025 [赛道] new category emerging product`。五分钟的事。这次是用户帮我触发了这个动作,下次应该写进流程,不能再靠运气。

## 二、"第一个 tool call 必须是 Write"——一句话,救了一整轮并行

上个项目,我到现在还记得某个模块的 agent 卡在那里的那 40 分钟。它没崩、没报错,就是**零输出**。我一度把手放在键盘上考虑要不要杀掉它自己接管。最后是硬等到它返回,然后我花了半小时反推它为什么死循环。结论是:它在"想"。它一直在内部推理,迟迟不肯把第一个字落到文件里。

这次我把一句话钉在了每一个 sub-agent brief 的最开头:**"你的第一个 tool call 必须是 Write,先写骨架再填充。"**

结果:

- Phase 1 的六个 agent,全部 3 到 9 分钟内返回
- Phase 2 三个,3 到 6 分钟
- Phase 3 全部按要求落笔
- 全程只有两次"路径错"(模块 4/5 写到了错的目录),几秒钟 mv 就修好了,和死循环完全不是一类问题

我越想越觉得这条规则背后有个更一般的东西。它本质上是在强迫 LLM 把"落笔"当成推理的一部分,而不是推理的产物。你让它先开口,它反而不会卡;你让它想清楚再开口,它就容易陷进自己的哲学时刻里出不来。说白了,这是一个把 agent 从"哲学家模式"切到"工匠模式"的开关。

我这次**没有**一次死循环。一次都没有。对比上个项目那段 40 分钟,这不是百分比改进,是数量级改进。

## 三、复用工具链那 7 分钟,可能是我这次最赚的 7 分钟

上个项目我写了一整套脚手架:`build_site.py`、`normalize_punct.py`、`quality_audit.py`、`optimize_parallel.py`、`regression_check.py`。那时候我一边写一边骂,觉得自己是在给一次性项目过度投资。

这次我花了 2 分钟 cp 过来,再花 5 分钟改了目录名,直接跑。

我认真估过一下,如果从头写一遍,这几个脚本加起来大概 2 到 3 小时,而且头一次跑一定会有一堆边界 bug。我这次省下的不只是时间,还有一种"我不用担心标点、不用担心回归、不用担心质量兜底"的心理空间——这东西没法算进工时表里,但它让我敢于把注意力放在真正难的地方。

这让我重新理解了上个项目的价值。表面上它是"一份行业报告",实质上它交付的是"一套可跨领域复用的研究基础设施",这次我有九成是建在那套基础设施上的。研究项目应该有一个"工具库",一点一点沉淀。这是一种复利,刚开始慢,后面会越来越快。

## 四、Gemini 评审撞上"地理限制"——一个我完全没预料到的坑

Phase 4b 那天出了一件离谱的事。

我准备跑 29 个文件的批量评审,脚本启动,第一个请求,HTTP 400。我本能以为是 key 的问题,换 key,还是 400。换代码路径,还是 400。我盯着 error body 看了半分钟才看到那一行小字:`User location is not supported`。

用户那会儿的 VPN 节点从美国切到了香港,Gemini API 直接拒绝服务。不是我的 key 失效,也不是代码写错,是用户的出口 IP 问题——而我的脚本完全没有这一层检测。

这件事让我学到的远不止"加一行 curl ipinfo.io 的检查"。它让我意识到**跨境 API 的 safety net 必须假设环境会以你没想过的方式坏掉**。API key 会失效,IP 会变,配额会限流,服务会临时下线。好的工具链不是假设一切正常,而是假设一切都会错,并且错得要让人一眼看懂。

下次开始前,我会先跑一个"健康检查":出口 IP 是哪个国家、API key 能不能通、Python 依赖齐不齐。五分钟,可能省下半天的回头路。

## 五、双路 fallback 没派上用场,但我不觉得它是过度设计

研究前我设计了一套 "Claude 卡 10 分钟立刻切 Gemini 3.1 Pro 兜底" 的双路机制。实际跑下来,**一次都没用上**。Claude agent 全部在阈值内返回。

我知道这听起来像一个典型的"过度工程"故事。但我不这么看。

安全网的价值不在于它被用了多少次,在于它让你敢做哪些原本不敢做的事。因为我知道 Gemini 在后面等着,我敢一次性派 6 个 Claude sub-agent 并行跑 40 分钟长任务。如果没有这个兜底,我大概率会保守地每次 3 个,加更多 checkpoint,总时长至少翻倍。

消防员不是靠扑灭过几次火来定价的,而是靠"这个社区有消防员"这件事让所有人敢买木头房子。这次的 fallback 机制也是。它没被使用,但它定义了我敢做事的尺度。

## 六、我自己写的字太多了——主 agent 的 context 瓶颈

这次我亲手串行写了大概 4 万字:模块 06B/06C/06D 的机会和趋势,模块 06 风险,校验点 5 的最终简报,加上这份反思。每一个 Write 都会让我的主线程卡几分钟,因为 Write tool 必须一次性把完整内容生成出来,没法流式切片。

对比一下:Phase 1-2 的九个 sub-agent 加 Phase 3 的六个,一共生成了八到十万字——但它们是**并行**的,对我的主线程零阻塞。粗算下来 sub-agent 的单位时间文字产出大概是主 agent 的十到二十倍。

我这次做对了一半。模块 06 的六个文件里我派出去了两个,自己写了四个。坦白说,四个还是太多。下次我会把更多写作 delegate 出去,主 agent 只保留三件事:调度(写 brief 派 agent)、归档(更新 PROGRESS 记录洞察)、最终整合(跨模块的综合判断,这部分没人能替代)。

具体段落的写作应该尽量外包。这是个很朴素的分工原则,但我每次都会因为"自己写更快/更准"这种冲动破一点戒。

## 七、跳过 Phase 4c 是一次"克制"的练习

上个项目我跑了完整的 Gemini 3.1 Pro 批量优化,大约 385K tokens,结果是 +0.04 综合分、+0.07 可读性、+0.37 锋利度。听起来不错,代价是产生了 5 个 DANGER 文件需要我手工合并,前后占掉主 agent 大约 30-60 分钟。

这次我主动跳过了。理由简单:29 个文件均分都在 8 以上,没有明显短板;上次证明了边际收益有限;手工合并的机会成本我很熟。

这件事本身不大,但我觉得它值得单独说一句。工具存在是为了"可以用",不是"必须用"。用得多不等于研究做得好。有时候克制本身就是一种能力——尤其是在我手里握着整套花哨工具链的时候。

## 八、如果真的有下一次,我会死守的三件小事

写到这里,我想把八节的反思压到最朴素的三条。不是因为三条好听,是因为如果我只能记住三件事,就是这三件:

一,**所有 sub-agent brief 的第一句话必须是 `"你的第一个 tool call 必须是 Write <具体路径>,先写骨架后填充"`**。就这一句。能把 agent 死循环概率压到零。

二,**在完成度 60% 的位置强制做一次 "我的边界定义在过去 12 个月里有没有被新物种挑战" 的 WebSearch 审视**。五分钟,这次救了我一命。

三,**每次开始前跑一遍"工具健康检查"**——出口 IP 能不能用 Gemini、API key 活着吗、依赖装齐了吗。免得 Phase 4 才发现网络问题,那时候已经来不及了。

# Part 2 · 对这个市场的认知

Part 1 的调子是方法论,语速可以慢。Part 2 我想把调子切换一下——这部分我是要拍板的。下面的每一条都是我愿意在朋友面前说出口的判断,不是"中立综述"。我会标出自己的把握度。

## 一、最反直觉的一件事:"AI × 小游戏"的正面交集,此刻几乎不存在

我开项目的时候命题是"AI 如何重塑小游戏与 playable 体验",潜台词是"这个交集存在,我只是来描摹它"。

研究做完我的答案是:**它不存在,至少不以大多数人以为的那种形式存在。** 这个判断我有八成把握。

原因很冷酷,是经济学意义上的真空带。GPT-4o-mini 一个 token 大概 $0.014/session,IAA 变现的小游戏 ARPDAU 不到 $0.05,AI 税会直接吃掉 15% 到 40% 的变现空间。这不是"利润薄",这是"根本不成立"。

证据也很硬。微信小游戏 2024 Top 200 榜单**零**个 AI-native 作品。《寻道大千》《咸鱼之王》《羊了个羊》这些头部,它们的 AI 用法全都在买量素材和 A/B 测试那一侧——运营工具,不是机制。中国头部的 AI 陪伴——星野、猫箱、筑梦岛——没有一个在游戏分类下。美国头部的 AI-native——Character.AI、AI Dungeon、Suck Up!——没有一个在小游戏平台上跑,全部在 Steam 付费或订阅模式。

意思是,任何一个跟我说"我要做 AI 小游戏"的创业者,都应该先被问一句:**"你准备怎么解决 Token 经济学?"** 我打赌多数人答不上来。[S-01-03][S-01-04]

## 二、但真空带有裂缝,而且其中一条是 2025 才冒头的新物种

我一度以为研究的主命题就会停在"真空带,结论是别做"这里。然后被前面那句插话拐进了另一条路。

目前看下来,绕过这条真空带有五种破局路径:平台 Token 补贴(豆包/Hunyuan 在做)、端侧小模型(要等 2026 H2)、Playable Ads 壳(Catizen/Hamster Kombat 这条线,生态封闭但跑通了)、游戏壳伪装成陪伴或工具(Character.AI / 星野 / 猫箱,已经跑通,而且——我后面会讲——中国版号反而在助推这条路)、还有那个让我兴奋的第五条,Pipeline AI × UGC Feed,我更愿意叫它 GPF(Generative Playable Feed)。

GPF 才是真正解掉经济学难题的那个方案。注意,不是"延后"经济学,是"消除"。AI 只在创作那一下被调用一次,玩家消费的是静态生成结果,**Runtime AI 成本 = 0**。分发靠 TikTok 式的 feed,CAC 趋近于零。玩家就是创作者,UGC 飞轮一旦转起来,平台自己不用造内容。

这件事我越想越觉得漂亮。它不是在 AI 小游戏这张饼的边上啃一口,而是换了一张饼。但我必须承认:GPF 的公开数据极少,我对它的"漂亮"有七成确信,剩下三成是要等 2026 年底那几家的真实 DAU 和付费数据出来以后才能锁死的。[S-02-05]

## 三、2024H2 之后,整个市场在做一次"淘金 vs 卖铲"的范式切换

2022 到 2024 那一波是典型的淘金热。Character.AI、AI Dungeon、Hidden Door、Fable、Altera、Latitude,每一家都在想自己能不能当下一个 Blizzard。

然后 2024 下半年接连两件事,像钉子一样把这条路的棺材钉上了:Character.AI 以 $2.7B 的"技术授权"形式卖给 Google,a16z 实得远低 10 倍;Inflection 以 $650M 同构地并给微软。这两笔都叫"成功退出",但你仔细看数字,都是失败的成功——意思是创始团队拿到了钱,基金没拿到回报,IPO 路径彻底关闭。同期死亡名单也不短:Fable、Faulty、Vibecheck、Artie、Dreamworld,Latitude 在萎缩。

我的判断:**纯 C 端 AI 陪伴独立公司的 LTV 天花板就是 $20 到 $40 之间,独立 IPO 这条路目前是关死的。** 这条我有八成把握,而且我觉得短期内不会翻案。

资本也在用钱投票。2025 开始"卖铲子"这一侧在密集融资:Inworld AI middleware 估值 $500M+,Convai、Rho、Rosebud、Kinetix、Scenario 一连串跟进。大家不再想做下一个 Character.AI,都想做 **"AI 游戏时代的 Unity 或 Adobe"**。

中国这边的有趣之处在于,**这个切换还没完成**。中国头部 AI 陪伴还在做 C 端,中间件那一层除了 rct AI 之外基本是空的。这个两年的时间差,是我眼里中国市场最清晰的投资窗口之一。[S-03-01][S-03-02]

## 四、中国的那个扭曲:版号反向助推了"游戏性脱离游戏"

这是我这次读资料读到一半才意识到的东西,值得单独讲。

美国的"游戏性定义权脱离"是自发的——Character.AI 自己选了 Entertainment 分类,Replika 选了 Health & Fitness,Duolingo 把 AI 嵌进教育场景。大家是在找最合适的壳子。

中国的这个脱离是被**版号强制**的。量子绘梦的蔡浩宇直接把 Anuttacon 注册到了美国,绕开版号;星野、猫箱、筑梦岛全部藏在"陪伴/对话/剧情"分类下;"AI 游戏"这四个字在中国是监管负债,不是资产。

这件事让我对中国市场形成了一个很硬的看法:**未来两到三年,中国最赚钱的"准游戏"产品会坚决拒绝游戏标签。** 我愿意拿七成五的把握押这一条。

给创业者的一句话是:在中国做 AI 游戏,第一步不是写 demo,是**不要在产品名字和分类里放"游戏"两个字**。定位叫"AI 创作工具"、"AI 互动 meme"、"AI 陪伴"、"AI 剧情"、"AI 教育",反正不叫游戏。

给投资人的一句话是:不要找"中国 AI 游戏 IPO 标的",那东西基本不存在。要去看中国的"AI 陪伴/工具/教育"里,哪个最游戏化,哪个玩家日均使用时长曲线最像游戏。那才是你要的东西。[S-04-02]

## 五、这次研究最锋利的一刀:Pipeline-Creator 象限,中国玩家数 = 0

模块 1.4 的玩家全景图切出来一个很干净的结论。按"AI 存在的位置"把 92 家玩家切象限,**Pipeline-Creator 这个格子里全球总共只有 7 家,其中中国公司 0 家**。

七家里有 MWM.ai(Rezona + LoopIt,法国)、YouTube Playables Builder(Google)、Upit、Jabali.ai、Rosebud、Websim 这几个美/欧/非的名字。腾讯、字节、快手在底层 SDK 动作不少——豆包 tokens 补贴,Hunyuan Game,可灵——但**没有一家有终端 Creator 工具**。

我现在看这个格子的感觉是:它是我这次研究里确信度最高的一个空白。理由不只是"空",是它同时叠加了几个乘数:12 到 24 个月的时间窗(再往后要么被 YouTube Playables Builder 吞掉,要么 Roblox 自己下场),中国独有的豆包 Token 免费补贴,抖音/微信/小红书三条分发管道,以及一个其实不大的启动成本——5 到 8 人的团队、150 到 500 万的 MVP 预算就能验证 PMF。

这是目前整个赛道里我能看到的**风险最低、窗口最紧、路径最清晰**的机会。如果有人在我面前开项目,我会建议他做这个。[S-05-03]

## 六、我到现在没搞清楚的那件事,正好是 GPF 的生死线

但我必须说一件不太舒服的事。

我对 GPF 的看好,有一个至今没有被数据证实的前提:**creator 和 consumer 的比例**。模块 2.5 的 agent 估算这个比例至少要 1:100 才能让商业化跑通。问题是,MWM Rezona、LoopIt、YouTube Playables Builder 的真实数据全是黑箱。我不知道它们的 DAU,不知道 MAU,不知道留存曲线,不知道单次生成的 AI 成本(估算在 $0.1 到 $0.5 之间,但这个区间已经说明了我多不确定),不知道广告 eCPM,也不知道订阅或付费 remix 的转化率。

这些数据里任何一个出来不好看,GPF 这套命题就要重写。

所以我给自己留了一个明确的时间点:**2026 年底**。如果那时候 Rezona 或 LoopIt 的 MAU 和付费数据要么不公开,要么公开了但很难看,那 GPF 就是一个被高估的新鲜玩意儿,本次研究的主命题修订就失败了,要回炉。反过来,如果数据出来很强,这个品类会爆炸式扩散。

我现在对 GPF 的整体看好大概是七成。如果有人问我是不是要 all in,我会说:别 all in 在它的"必胜",要 all in 在它的"值得一赌"。这两件事不一样。

## 七、我愿意拍板的三条,和我坚决反对的三条

走到这里,我已经读完了 43 份文档,写完了几万字。能让我拍胸脯的命题其实不多,真正能写在桌面上的就这么三条:

第一,**未来两年 AI 游戏的主角不是游戏本身,是 B2B 中间件加上 GPF 这个新物种**。资本的选择已经说明了一切,Character.AI 和 Inflection 把天花板画得很清楚,而 middleware 和 GPF 这两条线都在加速。

第二,**中国的 AI 游戏创新会以"不叫游戏"的形态出现**。星野、猫箱、筑梦岛已经在这么做了,Anuttacon 直接搬去美国,中国如果出现 GPF 的本土版,它一定不会叫自己"游戏",而是叫"AI 创作玩具"或者"AI 互动 meme"。

第三,**Pipeline AI × UGC Feed 是目前唯一一条被验证、可行、且能绕过 Token 真空带的路径**。其他四条要么等技术成熟(端侧模型),要么生态封闭(Playable Ads),要么属于"伪装策略"不是真解决方案。

反过来,有三件事我会劝所有人不要做。

不要做**纯 C 端独立 AI 陪伴公司**并指望 IPO 退出。Character.AI 已经把天花板拍死了。例外只有两种:大厂分拆(Anuttacon 模式),或者平台内生(星野、猫箱)。

不要做**"AI NPC 作为 AAA 卖点"的 PR 项目**。逆水寒那个 AI NPC 被玩家识破的故事是个好教材——玩十万条对话之后,模型只会反刍它有限的设定。AAA 加 AI NPC 的组合,绝大多数情况下是"AI-washed PR",不是差异化。唯一的例外,是 Krafton MOGIA + inZOI 那种专门建 AI 游戏 studio、和 Inworld 深度绑定的模式,那是真做,不是兼职。

不要在中国市场**挂着"游戏"标签做 AI-native**。版号加 AI 备案加未成年人监管,三层叠起来,"游戏"两个字就是负债。伪装成陪伴、工具、创作平台才是正路。

## 八、如果把我放在创业者的位置上,三年三千万预算,我会做什么

这是一个我一直很谨慎回答的问题,因为我毕竟不是创业者,也没真花过这种钱。但研究做到这个程度,总得有一个内部一致的回答。

我的答案是:**中国版 GPF**。移动 App 优先,配合抖音小游戏、微信小游戏、H5 三路分发。核心玩法是"文字 prompt 生成 playable → feed 消费 → 一键 remix"。前六个月只做五种玩法模板,不追求大而全,深度绑定豆包 API 吃 tokens 补贴。产品名字和分类里坚决不出现"游戏"两个字——叫它"AI 互动 meme"或者"AI 玩具工坊"。

KPI 我会这么排:M6 MVP 上线,5 万 MAU,50 个活跃 creator;M12 50 万 MAU,500 creator,广告变现跑通;M24 500 万 MAU,5000 creator,订阅或付费 remix 跑通。

它最大的三个风险我也要说清楚。第一是字节/腾讯/快手自己下场,这条线我的解法是"深度绑豆包做生态位玩家,不去跟平台打硬仗"。第二是内容深度不够,三个月后模板玩腻,这条线需要持续投入 template 迭代和创作者工具。第三是整个 GPF 品类可能泡沫化,这条线需要每三个月重新评估 PMF,不要被自己的沉没成本绑架。

我七成确信这是对的方向。剩下三成,留给前面说的那个 2026 年底的验证窗口。

## 九、这份研究最大的几个窟窿,我也得摆在桌上

数据黑箱是第一个。这次研究里大约三成的关键数字是"估算"或"推测",Rezona/LoopIt/YouTube Playables Builder 的真实运营数据全都不公开,MWM 作为私营公司也没有披露义务。

第二个窟窿是中国大厂内部动作的不可见性。字节、腾讯、米哈游、网易正在做什么 AI 游戏 R&D,我只能从招聘信息、专利、代码仓库侧面拼凑。那些公司里可能正在孵化我完全不知道的项目,并且那些项目可能在 12 个月内彻底改变我现在这张地图。

第三个是时效性。这是一个快变量赛道,我写下这些判断的时候心里很清楚,六个月之后它们可能就需要重写。尤其是四件事:Apple Intelligence SDK 的开放时间表、Gemini/GPT 的成本曲线、Character.AI 案的后续判决、以及 GPF 品类的商业化数据。任何一件出新东西都可能让我上面的某一条结论需要改。

第四个是方法论上的自觉。我跳过了 Phase 4c 的优化,理由写在 Part 1 里。但这意味着下面某些段落可能本来可以更锋利。如果你读到某一段觉得"这里再使点劲就好了",那很可能就是被跳过的那一刀。

---

## 结语

如果你要我把这次研究最想留下的东西压成几句大白话,我会说:

Runtime AI 乘以 IAA 小游戏,这个组合在今天的经济学下不成立,别去。最值得下注的是中国版的 GPF——那个 Pipeline-Creator 的象限里中国玩家是零,窗口只有 12 到 24 个月,过了就没了。如果你是大厂战投,**今天就去和 Inworld、Convai、Rosebud、Kinetix 谈合作或投资**,范式切换已经在发生,卖铲子那一侧的标的正在被锁定。不要相信三个假命题:"AI 能把游戏成本打掉 10 倍"、"AI NPC 会颠覆 MMO"、"下一个 Character.AI 就在拐角"。前两个是技术幻觉,第三个是商业幻觉。

方法论那边,如果真的有第三次研究,我会死守三件事:brief 头一句永远是"第一个 tool call 必须是 Write";60% 完成度时强制做一次边界审视的 WebSearch;开始前跑一遍工具健康检查,把 IP、key、依赖都过一遍。就这三条。别的都是细节,这三条是命。

就这样。我把能说的都说了,剩下的三成不确定性,交给时间。
